# CampOsprey Server ‚Äî Design Spec

**Date:** 2026-05-12
**Status:** Approved

## Overview

A Next.js full-stack server at `server/` that provides auth, trip storage, email notifications, and a future admin dashboard for the CampOsprey Chrome extension. The extension handles all BC Parks scanning; the server handles persistence, identity, and communication.

## Architecture

```
Extension (chrome.storage.local)
    ‚îÇ  fetch + Authorization: Bearer <token>
    ‚ñº
server/ (Next.js on VPS, nginx reverse proxy)
    ‚îú‚îÄ‚îÄ /api/auth/[...all]        ‚Äî better-auth (emailOTP + bearer + admin)
    ‚îú‚îÄ‚îÄ /api/trips                ‚Äî CRUD, auth-required
    ‚îú‚îÄ‚îÄ /api/trips/:id/result     ‚Äî booking outcome ‚Üí Resend email
    ‚îú‚îÄ‚îÄ /api/user                 ‚Äî current user profile
    ‚îî‚îÄ‚îÄ /admin                   ‚Äî dashboard (gated by admin role)
    ‚îÇ
    ‚ñº
Postgres ‚Äî database: parkbooking (VPS/AWS)
```

**Key decisions:**
- **Bearer tokens** ‚Äî chrome extensions are cross-origin; token stored in `chrome.storage.local`, sent as `Authorization: Bearer <token>`. better-auth `bearer` plugin handles this.
- **Passwordless OTP only** ‚Äî email ‚Üí 6-digit code ‚Üí session token. No passwords.
- **Extension owns scanning** ‚Äî server never polls BC Parks. The extension pushes results when a booking attempt completes.
- **Server-primary trips** ‚Äî Postgres is the source of truth. Extension syncs on login and after every mutation.

## Auth Flow

**Sign up and sign in are the same passwordless flow:**

```
1. Extension sends email   ‚Üí POST /api/auth/email-otp/send-verification-otp
2. Resend delivers 6-digit OTP (expires 5 min)
3. Extension sends email+OTP ‚Üí POST /api/auth/email-otp/verify-email
4. Server returns { token, user }
5. Extension saves token to chrome.storage.local
```

User is created automatically on first successful OTP verify ‚Äî no separate sign-up step.

**better-auth plugins:**
- `emailOTP` ‚Äî send + verify, auto-creates user on first verify
- `bearer` ‚Äî accepts `Authorization: Bearer <token>` (required for extension cross-origin calls)
- `admin` ‚Äî role-based access for admin dashboard

**Public endpoints** (no auth required):
- `POST /api/auth/email-otp/send-verification-otp`
- `POST /api/auth/email-otp/verify-email`

**Auth-required endpoints:**
- `GET /api/trips`, `POST /api/trips`
- `PUT /api/trips/:id`, `DELETE /api/trips/:id`
- `POST /api/trips/:id/result`
- `GET /api/user`

**Token lifecycle:** 7-day expiry, refreshed on activity. Extension shows a reconnect prompt in the popup when the token expires.

## Trip Sync Model

The server is the source of truth. The extension keeps a local copy in `chrome.storage.local` for offline operation.

**On login:** `GET /api/trips` ‚Üí replace local storage entirely.

**On user-initiated CRUD:**
- Create: `POST /api/trips` ‚Üí on success, add to local storage
- Update: `PUT /api/trips/:id` ‚Üí on success, update local storage
- Delete: `DELETE /api/trips/:id` ‚Üí on success, remove from local storage

**Scanner updates** (status, lastMatch, attempted) are pushed via `PUT /api/trips/:id`:
- On every status transition
- When `lastMatch` is set
- Every 10 new `attempted` entries (batched to avoid hammering the API)

**Offline behaviour:** Extension operates on local storage as fallback. Failed writes are queued and retried on the next successful request.

**Multi-device:** A second device logs in, fetches trips, and is immediately in sync. If a trip shows `status: scanning` from another device, the second device skips it to avoid duplicate scanning.

## Notification Flow

```
Extension completes a booking attempt
    ‚îÇ
    POST /api/trips/:id/result  { outcome, matchedSite, error? }
    ‚îÇ
    ‚ñº
Server:
  1. Inserts row into booking_results
  2. Updates trip status in trips table
  3. Sends email via Resend
  4. Sets emailSent = true
```

**Outcomes and email subjects:**

| outcome | Email subject |
|---|---|
| `found` | "Campsite found at {parkName}" |
| `hold_placed` | "Campsite held ‚Äî complete your booking" |
| `booked` | "Campsite booked! You're going camping" |
| `failed` | "Booking failed at {parkName}" |

## Admin Dashboard

Gated by better-auth `admin` plugin (user must have `role: 'admin'`). Initial scope:
- List all users with trip counts and last active date
- List booking results across all users

Full admin feature set is out of scope for this implementation ‚Äî the layout and auth gate are scaffolded now, content expanded later.

## Deployment

- **Runtime:** `next start` on VPS
- **Reverse proxy:** nginx (HTTPS termination, proxy to `localhost:3001`)
- **Database:** Postgres `parkbooking` database on existing VPS/AWS infrastructure
- **Email:** Resend
- **Environment variables:** `DATABASE_URL`, `BETTER_AUTH_SECRET`, `RESEND_API_KEY`, `APP_URL`
